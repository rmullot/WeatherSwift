## Project

WeatherSwift is an iOS weather app (Swift, UIKit, iOS 12.1+) that fetches GFS forecasts from the infoclimat.fr public API, caches them in Core Data, and displays them in a table with a detail view. CocoaPods is used for dependencies. The repo root is `WeatherSwift/` (the Xcode project lives one level below the git root, in `WeatherSwift/WeatherSwift/`).

## Commands

All commands run from `WeatherSwift/` (the directory containing `WeatherSwift.xcworkspace`).

- Install deps: `pod install`
- **Always open/build the `.xcworkspace`, not the `.xcodeproj`** (CocoaPods).
- Build: `xcodebuild -workspace WeatherSwift.xcworkspace -scheme WeatherSwift -sdk iphonesimulator build`
- Run all tests: `xcodebuild -workspace WeatherSwift.xcworkspace -scheme WeatherSwift -destination 'platform=iOS Simulator,name=iPhone 15' test`
- Run a single test: append `-only-testing:WeatherSwiftTests/ForecastViewModelTest/testName` to the test command (replace bundle/class/method as needed).
- Lint: SwiftLint runs automatically as an Xcode build phase. Config is `.swiftlint.yml` (disabled: `line_length`, `force_cast`; opt-in: `empty_count`; Pods excluded).

## Code generation (SwiftGen)

Localized strings are code-generated. **Do not hand-edit `WeatherCore/Generated/Strings.swift`** — it is regenerated by SwiftGen from `WeatherCore/Resources/en.lproj/Localizable.strings` per `swiftgen.yml` (runs as a build phase). Add new strings to the `.strings` files (`en.lproj`, `fr.lproj`); the project is localized English/French.

## Architecture

Two build targets separate reusable logic from UI:

- **WeatherCore** (dynamic framework): all networking, parsing, persistence, and domain services. Contains no view code. Public API is exposed via `public` symbols consumed by the app target.
- **WeatherSwift** (app): MVVM presentation layer under `WeatherSwift/MVVM/` (`Views/` + `ViewModels/`), plus a `NavigationService`. UI is storyboard-driven (`Main.storyboard`).

### Data flow

The forecast pipeline is a chain of singleton services (most are `sharedInstance`):

1. `BridgeForecastService.getForecasts` — the single entry point the app calls. Returns cached Core Data `Forecast` objects immediately, then (unless offline) refreshes from the network and returns updated results.
2. `WebServiceService` — builds the infoclimat URL, requests location permission via `PermissionService` (forecast is fetched for the device's GPS coordinates), fetches JSON. Carries an `onlineMode` (`.online`/`.offline`) flag that gates network refreshes.
3. `ParserService` → `Forecast+Parsing` — parses JSON into `ForecastStruct` value types.
4. `CoreDataService` — `clearData()` then `saveForecastsData()` to replace the cache; `saveContext()` is called on app terminate.

### Core Data

The model is `WeatherSwift.xcdatamodeld`. `Forecast` is the managed object (`Forecast+CoreDataClass/Properties`), with a generic fetching layer in `Models/Fetchable/` (`Fetchable` protocol + `objectsInContext(_:predicate:sortedBy:)`).

### Networking utilities

`WeatherCore/Network/` holds connectivity plumbing: `ReachabilityService` (via ReachabilitySwift pod) toggles online/offline behavior, `NetworkActivityService`, a `MultiCastDelegate`, and a mutex-based counter (`MutexCounter` / third-party `CwlMutex`).

### UI testing hook

The app checks launch arguments/environment (see `UITestKey.swift`) so UI tests can force offline/mock behavior. When adding UI-test-controllable behavior, wire it through this key rather than ad-hoc flags.

## Dependencies (CocoaPods)

SwiftLint, SwiftGen, SwiftMessages (in-app banners/alerts), ReachabilitySwift. Both `WeatherSwift` and `WeatherCore` targets link ReachabilitySwift + SwiftMessages.
